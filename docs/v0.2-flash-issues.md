# Flash Pipeline Issues — v0.2

Issues discovered during a real end-to-end flash session (2026-04-13).

## Resolution: delegate to ZSA's `zapp` CLI

ZSA released [`zapp` v1.0.0](https://github.com/zsa/zapp) on 2026-04-07 —
a cross-platform flasher that owns the USB DFU and HALFKAY protocols
natively via libusb and supports every ZSA board we care about
(Voyager, Moonlander rev A/B, Ergodox EZ STM32/Teensy, Halfmoon,
Planck EZ).

v0.2 replaces oryx-bench's in-tree `wally-cli` / `dfu-util` / Keymapp
backends with a single `zapp` handoff (`src/flash/zapp.rs`,
subprocess). This restores `ARCHITECTURE.md`'s long-standing non-goal
("we never invoke `dfu-util` directly") and collapses three divergent
flash paths into one.

Issues #1–#4 below are resolved by that adoption. Issues #5–#8 are
unrelated to flashing (they're Docker-build issues we hit during the
session) and are kept for tracking.

## 1. ~~wally-cli does not support the Voyager~~ [RESOLVED by zapp]

**Previous severity:** Critical — prevented flashing entirely.

**Was:** `wally-cli` polls for `0x0483:0xDF11` (STM32 DFU), but the
Voyager bootloader enumerates as the ZSA-specific `0x3297:0x0791`, so
wally-cli never detected it.

**Resolution:** `zapp`'s board dispatch reads the device's USB ID and
speaks the right protocol natively. No wally-cli involvement anywhere
in the pipeline.

## 2. ~~DFU suffix incompatibility with wally-cli~~ [RESOLVED by zapp]

**Previous severity:** Medium.

**Was:** wally-cli v2.0.x has a byte-order bug parsing the DFU suffix
(`expected 0x83:0x11 got 0x97:0x91` — reading high bytes of the
16-bit vendor/product words).

**Resolution:** zapp parses the suffix correctly — no byte-order bug.

**Important correction (2026-04-15):** an earlier draft of this doc
claimed zapp ignored the DFU suffix and identified devices purely at
the USB protocol level. That was wrong. zapp's loader requires *either*
a DFU suffix (for STM32 .bin) or an Intel HEX header (for HALFKAY .hex)
and rejects raw .bin with:

```
Invalid firmware: not a valid firmware file
(no DFU suffix or Intel HEX header found)
```

We briefly disabled `DFU_SUFFIX_ARGS` in the generated `rules.mk`
based on the incorrect assumption; every v0.2.0 firmware produced
this way is un-flashable by zapp. The override has been removed so
QMK's upstream Voyager `DFU_SUFFIX_ARGS` (correct vendor/product IDs)
stamps the suffix during compile.

## 3. ~~`Auto` backend detection doesn't account for board type~~ [RESOLVED by zapp]

**Was:** `detect_backend` picked whichever backend was on PATH
regardless of board — wally-cli worked for Moonlander/Ergodox but
failed for Voyager.

**Resolution:** `zapp` handles per-board dispatch internally. Our
`detect_backend` now only checks that `zapp` exists on PATH and is at
the required minimum version; all board-specific logic lives in zapp.

## 4. ~~udev rules for Voyager bootloader not documented~~ [RESOLVED by zapp]

**Was:** `dfu-util` failed with permission denied for the Voyager's
`3297:0791` bootloader without a custom udev rule.

**Resolution:** zapp ships `udev/50-zsa.rules` upstream — point users
at `https://github.com/zsa/zapp#linux-udev-rules` rather than
documenting a rule in two places. oryx-bench no longer maintains its
own copy.

## 5. Docker build output `cp` permission denied

**Severity:** Medium (build fails, firmware is actually produced)

**Symptom:**
```
cp: cannot create regular file 'zsa_voyager_oryx-bench.bin': Permission denied
```

**Root cause:** QMK's Makefile copies the `.bin` from `.build/` (tmpfs, writable) to `/firmware/` (root-owned, not writable under `--user`). Make exits non-zero even though the `.bin` was successfully produced.

**Fix applied (v0.1):** Wrapper bash script in `docker.rs` checks if `.bin` exists in `.build/` after `qmk compile`, copies it to `/work/` (bind-mounted project root), and exits 0.

**Proper v0.2 fix:** Same approach, but consider adding a `--output-dir` flag to the QMK compile invocation if ZSA's fork supports it. If not, the bash wrapper is the correct solution — it's not a workaround, it's an adapter for QMK's assumption that it owns its working directory.

## 6. Docker image missing `dfu-util` package

**Severity:** Low (fixed in v0.1)

**Root cause:** Ubuntu 24.04 base image doesn't include `dfu-util`. QMK's Makefile calls `dfu-suffix` (from the `dfu-util` package) during the build.

**Fix applied:** Added `dfu-util` to `apt-get install` in the Dockerfile.

**Status (2026-04-15):** `dfu-util` remains required — `dfu-suffix`
is invoked during QMK build to stamp the DFU suffix that `zapp`'s
loader requires. The package is installed in the Dockerfile.

## 7. Git "dubious ownership" in Docker corrupts Makefile

**Severity:** High (build fails with cryptic "multiple target patterns" error)

**Root cause:** `/firmware` is owned by root (built into the image). Docker runs with `--user UID:GID`. Git warns about dubious ownership. The QMK Python wrapper's warning (`⚠ "git describe..." returned error code 128`) goes to stdout, which Make's `$(shell)` captures. The colon in `fatal:` is interpreted as a Make target pattern separator.

**Fix applied:** `git config --system --add safe.directory '*'` in the Dockerfile. The wildcard is safe — this is an isolated single-purpose build container.

## 8. Ubuntu 24.04 dash vs bash in Docker

**Severity:** High (build fails with "Syntax error: || unexpected")

**Root cause:** QMK's Makefile recipes use bash-isms (`||` in recipe lines). Ubuntu 24.04 defaults `/bin/sh` to dash, which doesn't support these constructs.

**Fix applied:** `ln -sf /bin/bash /bin/sh` in the Dockerfile.

## Summary — v0.2 Flash Pipeline

```
oryx-bench flash
  │
  ├── Verify .oryx-bench/build/firmware.bin is fresh
  ├── Render plan (target, size, sha256, via: zapp)
  ├── Ask for user confirmation (unless --yes)
  │
  └── exec `zapp flash <firmware.bin>`
        (inherited stdio: zapp's progress bar drives the user)
```

Single backend. Zero in-tree flash protocol logic. No dfu-util
subprocess. Non-goal #7 from `ARCHITECTURE.md` honored.
