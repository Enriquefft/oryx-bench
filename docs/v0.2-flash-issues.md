# Flash Pipeline Issues — v0.2

Issues discovered during a real end-to-end flash session (2026-04-13).

## 1. wally-cli does not support the Voyager

**Severity:** Critical — prevents flashing entirely

**Symptom:** `wally-cli <firmware.bin>` spins on "Press the reset button" indefinitely. When the reset button is pressed, `libusb: interrupted [code -10]` errors appear but the device is never detected.

**Root cause:** wally-cli was designed for the ErgoDox EZ and Moonlander. These boards enter bootloader mode as STM32 DFU devices (`0x0483:0xDF11`). The **Voyager uses a ZSA-specific bootloader** that enumerates as `0x3297:0x0791`. wally-cli polls for `0x0483:0xDF11` and never finds it.

**Evidence:**
```bash
# During bootloader mode:
$ lsusb | grep -i zsa
Bus 003 Device 023: ID 3297:0791 ZSA Technology Labs Voyager Bootloader

# NOT this (which is what wally-cli looks for):
# Bus 003 Device 023: ID 0483:df11 STMicroelectronics DFU in FS Mode
```

**Fix:** oryx-bench should use `dfu-util` directly for the Voyager. The correct command:
```bash
dfu-util -d 3297:0791 -a 0 -s 0x08002000:leave -D <firmware.bin>
```

The DFU parameters come from the Voyager's `rules.mk`:
```makefile
DFU_ARGS = -d 3297:0791 -a 0 -s 0x08002000:leave
```

**Implementation:** Add a `DfuUtil` backend to `src/flash/mod.rs` alongside `Wally` and `Keymapp`. The `Auto` detection should:
1. Check geometry — if Voyager, use `dfu-util` (not wally-cli)
2. Fall back to `keymapp` handoff if `dfu-util` not on PATH

The backend should invoke:
```
dfu-util -d <vendor>:<product> -a 0 -s <start_address>:leave -D <firmware.bin>
```

Where `vendor`, `product`, and `start_address` come from the `Geometry` trait — each board defines its own DFU parameters.

**Requires `sudo`/`doas`:** The DFU device needs root access unless udev rules grant user permissions. oryx-bench should detect this and suggest `sudo` when the device can't be opened.

## 2. DFU suffix incompatibility with wally-cli

**Severity:** Medium (blocked by #1, but independently broken)

**Symptom:** wally-cli rejects the firmware:
```
Error while extracting DFU Suffix: Invalid vendor or product id, expected 0x83:0x11 got 0x97:0x91
```

**Root cause:** wally-cli v2.0.x has a byte-order bug in DFU suffix parsing. It reads single bytes from the suffix instead of 16-bit little-endian words. `0x97` is the high byte of vendor `0x3297`; `0x91` is the high byte of product `0x0791`.

**Fix applied (v0.1 workaround):** The generated `rules.mk` overrides `DFU_SUFFIX_ARGS =` (empty), disabling suffix generation. The suffix serves no purpose in the oryx-bench pipeline — `dfu-util` validates devices via USB ID at the protocol level, not via file metadata.

**Proper v0.2 fix:** This workaround is correct and should remain. The DFU suffix is dead code in our pipeline. If a future backend needs it, add it back per-backend rather than globally.

## 3. `Auto` backend detection doesn't account for board type

**Severity:** Medium

**Symptom:** `Auto` selects wally-cli (first binary found on PATH) regardless of keyboard model. wally-cli works for Moonlander/ErgoDox but fails for Voyager.

**Fix:** `detect_backend` should take the `Geometry` as a parameter and select the correct backend per board:

```rust
pub fn detect_backend(requested: BackendChoice, geom: &dyn Geometry) -> Result<Backend> {
    match requested {
        BackendChoice::Auto => {
            match geom.flash_protocol() {
                FlashProtocol::ZsaDfu => { /* dfu-util with ZSA vendor ID */ }
                FlashProtocol::Stm32Dfu => { /* wally-cli or dfu-util */ }
            }
        }
        // ...
    }
}
```

## 4. udev rules for Voyager bootloader not documented

**Severity:** Low (NixOS-specific)

**Symptom:** `dfu-util` fails with permission denied unless run as root. The Voyager bootloader enumerates as `3297:0791` which needs a udev rule.

**Required udev rule:**
```
SUBSYSTEMS=="usb", ATTRS{idVendor}=="3297", ATTRS{idProduct}=="0791", MODE:="0666"
```

**Fix:** Document in README or `oryx-bench setup` output. Optionally, `oryx-bench setup --full` could check for this rule and warn if missing.

## 5. Docker build output `cp` permission denied

**Severity:** Medium (build fails, firmware is actually produced)

**Symptom:**
```
cp: cannot create regular file 'zsa_voyager_oryx-bench.bin': Permission denied
```

**Root cause:** QMK's Makefile copies the `.bin` from `.build/` (tmpfs, writable) to `/firmware/` (root-owned, not writable under `--user`). Make exits non-zero even though the `.bin` was successfully produced.

**Fix applied (v0.1):** Wrapper bash script in `docker.rs` checks if `.bin` exists in `.build/` after `qmk compile`, copies it to `/work/` (bind-mounted project root), and exits 0.

**Proper v0.2 fix:** Same approach, but consider adding a `--output-dir` flag to the QMK compile invocation if ZSA's fork supports it. If not, the bash wrapper is the correct solution — it's not a workaround, it's an adapter for QMK's assumption that it owns its working directory.

## 6. Docker image missing `dfu-util` package

**Severity:** Low (fixed in v0.1)

**Root cause:** Ubuntu 24.04 base image doesn't include `dfu-util`. QMK's Makefile calls `dfu-suffix` (from the `dfu-util` package) during the build.

**Fix applied:** Added `dfu-util` to `apt-get install` in the Dockerfile.

## 7. Git "dubious ownership" in Docker corrupts Makefile

**Severity:** High (build fails with cryptic "multiple target patterns" error)

**Root cause:** `/firmware` is owned by root (built into the image). Docker runs with `--user UID:GID`. Git warns about dubious ownership. The QMK Python wrapper's warning (`⚠ "git describe..." returned error code 128`) goes to stdout, which Make's `$(shell)` captures. The colon in `fatal:` is interpreted as a Make target pattern separator.

**Fix applied:** `git config --system --add safe.directory '*'` in the Dockerfile. The wildcard is safe — this is an isolated single-purpose build container.

## 8. Ubuntu 24.04 dash vs bash in Docker

**Severity:** High (build fails with "Syntax error: || unexpected")

**Root cause:** QMK's Makefile recipes use bash-isms (`||` in recipe lines). Ubuntu 24.04 defaults `/bin/sh` to dash, which doesn't support these constructs.

**Fix applied:** `ln -sf /bin/bash /bin/sh` in the Dockerfile.

## Summary — What v0.2 Flash Pipeline Should Look Like

```
oryx-bench flash
  │
  ├── Detect board from kb.toml geometry
  │
  ├── Voyager → dfu-util backend
  │     ├── Check dfu-util on PATH
  │     ├── Check udev permissions (try opening device)
  │     ├── Prompt: "Press reset button"
  │     ├── Poll for 3297:0791 USB device
  │     └── dfu-util -d 3297:0791 -a 0 -s 0x08002000:leave -D <fw.bin>
  │
  ├── Moonlander → wally-cli or dfu-util backend
  │     └── (TBD when Moonlander support ships)
  │
  └── Fallback → Keymapp GUI handoff
        └── Stage firmware + print instructions
```
